# 빌리조 AI 상세페이지 자동 생성 룰북

> **문서 목적**: billyjo.co.kr에 등록되는 모든 카테고리의 제품에 대해, AI가 사용자 페이지 진입 시 일관된 형식으로 상세페이지(종합점수 게이지 + 4지표 + 스펙요약 + 기본정보 + 주요 구독자 구성 + 상세스펙 + 예상 지원금·사은품)를 자동 생성하기 위한 카테고리별 룰북.
>
> **작성일**: 2026-05  
> **버전**: v0.1 (초안 — 정수기·에어컨 검증 완료, 나머지 카테고리 룰북 확장 예정)

---

## 1. 시스템 개요

### 1.1 작동 원리 요약

```
[제품 등록] → [카테고리 라우터] → [패밀리별 룰북 선택] → [공식 출처 크롤] 
   → [AI 추출+검증] → [점수화] → [DB 저장] → [GTM 트리거 시 캐시 조회 → 렌더링]
```

핵심 원칙은 다음 4가지.

- **카테고리 라우팅 우선**: 모든 제품은 등록 시점에 14개 카테고리 패밀리 중 하나로 분류되며, 각 패밀리는 자체 룰북을 갖는다. 룰북이 잘못 매칭되면 카드 내용이 통째로 어색해진다.
- **공식 출처 우선**: 제조사 공식 사이트와 공인 가격비교(다나와) 등 화이트리스트 출처에서만 정보를 가져오며, 모든 값은 출처 URL과 함께 저장한다.
- **사람 검수 필수**: AI가 추출한 값 중 신뢰도가 임계치 미만이거나 단일 출처에서만 확인되는 항목은 직원 검수 큐로 자동 이동한다.
- **사전 생성 + 캐싱**: 제품 등록 시 1회 생성해 DB에 저장하고, 사용자 페이지 진입 시에는 캐시만 읽어와 렌더링한다. 실시간 AI 호출은 없다.

### 1.2 공통 카드 구조 (전 카테고리 동일)

모든 카테고리의 카드는 동일한 7개 슬롯으로 구성된다. 슬롯 내부의 항목·점수·페르소나만 패밀리별로 다르다.

| # | 슬롯 | 역할 | 데이터 출처 | 위치 (v0.2.7+) |
|---|------|------|------------|---------------|
| 1 | 헤더 | 제품명, 모델번호, 형태 한 줄 요약 | 제조사 공식 | **Hero (.prod_view_top.right)** |
| 2 | 게이지 + 4지표 | 종합점수 + 4개 평가축 점수 | AI 점수 산정 | **Hero (.prod_view_top.right)** |
| 3 | **6-8칸 스펙요약** | 6 기술 스펙 + (선택) 2 운영/계약 스펙 | 제조사 + 다나와 + 빌리조 | AI 카드 본체 |
| 4 | 기본정보 | 제품 유형 분류 (pills) + 약정·위약금 (rental-terms) + 강점 칩 (strengths) | AI 추천 룰 + 빌리조 계약 | AI 카드 본체 |
| 5 | 주요 구독자 구성 (Top 3) | 페르소나 3종 + 구성비 % | 내부 DB + AI 매칭 | AI 카드 본체 |
| 6 | 상세 스펙 (Step 1~3) | 단계별 평가 항목 + pill 옵션 (+35% 폰트 스케일) | 제조사 공식 | AI 카드 본체 |
| 7 | 예상 지원금·사은품 | DB 지원금 + AI 예측 사은품 + 제조사 프로모션 | 내부 DB + AI 예측 | AI 카드 본체 |

**SLOT 1·2 Hero 재배치 (v0.2.7)** — SLOT 1(`.head .meta`)과 SLOT 2(`.summary` 게이지+4지표)는 `.prod_view_top.right`(이미지+가격 hero 영역) 안으로 이동해 첫 화면에 평가 결과까지 노출. AI 카드 본체의 `.head`/`.summary`/`.ai-card-tag`는 `display:none` 처리. 자세한 처리(layout flex 변환, CSS 변수 스코프 복원)는 CLAUDE.md 절대 규칙 #22 참조.

**SLOT 6 Step 폰트 +35% (v0.2.7)** — `.step-n` 26px / `.step-title` 16px / `.step-sum` 14px / `.step-details summary` 13px / `.grade-badge` 13px / `.pills .pill` 12.5px. 모바일(≤600px) 각 -1~2px. 자세한 CSS는 CLAUDE.md 절대 규칙 #23 참조.

**SLOT 6 카테고리 선택형 스펙 — 2단계 토글 비교표 (v0.2.9)** — `.step-details` 내부 스펙 중 1-of-N 카테고리형(필터 종류·살균 방식·컴프레서 등)은 pill 대신 `.field.spec-compare` 패턴 사용: (1) 라벨 + 선택 값 ("필터 종류: UF 중공사막") + sub-toggle pill, (2) sub-toggle 클릭 시 3-4 컬럼 비교표 전개. 선택 컬럼은 브랜드 파랑 강조(`.ft-on`), 비선택은 회색(`.ft-off`), 평가없음은 회색 채움(`.ft-na`). 다중 선택·수치형 스펙은 pill 유지. 자세한 마크업과 적용 후보는 CLAUDE.md 절대 규칙 #24 참조.

**하단 fixed 위젯 (`.bb-inner`) — v0.3.2** — `.prod_view_bot.card.mt40` 컨테이너를 **`position:fixed; bottom:0; left:0; right:0; z-index:99999`**로 viewport 하단에 진짜 고정 (v0.3.0 sticky → v0.3.1 fixed 정정 → v0.3.2 specificity·JS fallback 강화). 핸들(grip + 제품명 + 가격 + ▾/▴ 토글)로 펼침/접힘. 버튼 3종 표준: `[장바구니] [렌탈+사은품 신청 🎁 브랜드 파랑] [상담신청 💬 아웃라인]`. "렌탈 신청" 단독 명칭과 전화기 아이콘은 폐기. **모바일 헤더 (v0.3.2)**: `.new-gnb__wrap`·`.new-gnb`·`.bj-internet-li`는 ≤768px에서 명시 `display:none` — inject.js가 `.mobile__gnb`로 별도 노출하므로 PC GNB 카테고리 메뉴는 불필요. 헤더 `.header__top { gap:0 }` + 햄버거 `margin-right:5px` + 우측 아이콘 `margin-left:auto`로 [햄버거-5px-로고-자유공간-아이콘] 정렬 명시화 (이벤트 버튼이 로고와 겹치는 문제 해결). 자세한 CSS·JS·접근성 사양은 CLAUDE.md 절대 규칙 #21·#25 참조.

**SLOT 3 — 스펙요약 6-8칸 (v0.2.6+)**

- 기본 6칸은 **기술 스펙** (패밀리별 정의, 본 문서 §4 참조)
- **선택 7-8번째 칸은 운영/계약 스펙** — 패밀리 공통 후보 (모두 빌리조 본사 데이터):
  - **방문관리** (예: "4개월 코디", "2개월 본사관리") → `.sv` 안에 `.hq-btn` pill로 시각 강조 (🏢 + 브랜드 파랑)
  - **소유권 이전** (예: "60개월 후 자동 이전", "약정 종료 후 매입")
  - **A/S 보증** (예: "무상 6개월", "유상 연장 가능")
- PC 3-col grid · 모바일 (≤600px) 2-col 자동 wrap. 8칸이면 PC 3행 (마지막 행 2칸 좌측 정렬) · 모바일 4행
- **중복 흡수 원칙**: 같은 정보가 SLOT 3·SLOT 4 (강점 칩·rental-terms)·SLOT 5 (페르소나 멘트)에 동시 노출되지 않도록 단일화. 일반 우선순위는 **`.specs` > `.strengths` > `.rec`(폐기) > 페르소나 멘트** — 가장 구조화된 그리드에 우선 편입하고 산문 영역에서 제거. 폐기된 `.rec` 한 줄 산문 블록은 v0.2.6에서 폐지됨.

### 1.3 종합점수 산정 공식

```
종합점수 = (지표1 + 지표2 + 지표3 + 지표4) ÷ 4
```

4개 지표는 패밀리마다 다르지만, 산정 공식은 동일. 각 지표는 0~100점 스케일이며, 100점 만점 기준으로 게이지가 채워진다.

## 3. 패밀리별 룰북 템플릿

모든 패밀리 룰북은 다음 항목을 정의한다.

```yaml
family_id: F0X
family_name: "패밀리명"
applicable_categories: [세부 카테고리 리스트]
metrics_4:                       # 게이지 옆 4지표
  - { id: m1, label: "지표명", weight: 1.0 }
  - ...
spec_summary_6:                  # 6칸 기본 스펙 (기술)
  - { id: s1, label: "스펙명", source_field: "..." }
  - ...
spec_summary_ops:                # 선택 7-8번째 칸 (운영/계약) — 빌리조 본사 데이터 기준
  - { id: visit_care, label: "방문관리", source_field: "billyjo.contract.visit_cycle", style: "hq-btn", optional: true }
  - { id: ownership_transfer, label: "소유권", source_field: "billyjo.contract.ownership_after_months", optional: true }
  # 추가 후보: free_as (무상 A/S 기간) 등. 최대 2개까지만 노출, 그 외는 SLOT 4로 보낸다.

step_compare_specs:              # v0.2.9 — SLOT 6 Step 내부 카테고리 선택형 스펙 비교표
  - id: filter_type              # F01 예
    step: 1
    label: "필터 종류"
    options: ["역삼투압(RO)", "중공사막", "나노필터"]
    selected: 1                  # 0-indexed (이 제품의 옵션 인덱스)
    compare_axes:                # 3개 비교축
      - { label: "정수 속도", values: ["느림", "빠름", "빠름"] }
      - { label: "미네랄", values: ["미통과", "통과", "통과"] }
      - { label: "제거 효율", values: ["가장 좋음", "좋음", "매우 좋음"] }
    sub_toggle_label: "다른 필터와 비교"
    desc: "정수 성능은 필터 방식에 따라 차이가 큽니다."
  # 평가없음 케이스: selected=null, options 끝에 "평가없음" 추가, 해당 컬럼 ft-na 처리
step1: { title: "...", fields: [...] }     # 상세 스펙 Step 1
step2: { title: "...", fields: [...] }     # 상세 스펙 Step 2
step3: { title: "...", fields: [...] }     # 상세 스펙 Step 3
persona_pool: [페르소나 ID 리스트]          # Top 3 선정 대상
gift_patterns:                              # 브랜드별 사은품 패턴
  - { brand: "...", typical_gifts: [...] }
source_whitelist: [공식 사이트 URL 리스트]
```

## 4. 패밀리별 룰북

### F01. 정수·물처리

**적용 카테고리**: 정수기, 연수기/이온수기, 샤워기필터  
**검증 완료 사례**: 코웨이 CHP-7211N, CHPI-7400N_V2 / 쿠쿠 CP-F603SW / 세스코 EWBD351·EWBS351

**4지표**: 렌탈료 · 정수성능 · 위생관리 · 편의기능

**6칸 스펙요약**: 렌탈료 / 필터종류 / 필터개수 / 편의기능 개수 / 살균방식 / 제품유형

**Step 1. 정수성능**
- 정수 기능 (개수): pills `냉수 / 온수 / 정수 / 끓인물 / 얼음 / 탄산 / 살균수`
- 필터 종류: pills `RO 역삼투압 / 나노트랩 / UF 중공사막`
- 필터 개수: pills `1개 / 2개 / 3개 / 4개+`

**Step 2. 위생관리**
- 살균 방식 (개수): pills `UV살균 / 자동살균 / 고온살균 / 전해수살균`
- 살균 부위 (개수): pills `직수관 / 코크 / 냉수관 / 저수조 / 얼음저장고 / 얼음트레이 / 얼음파우셋`
- 직수관/저수조 재질: 텍스트 (스테인리스 / 트라이탄 / PP / 폴리케톤)
- 위생 기능: pills `분리세척 / 무빙코크 / 자동잔수비움 / 분리형 코크`

## 7. 출처 화이트리스트 (전체)

### 7.0 빌리조 자체 상세페이지 — 1순위에 준함 (v0.2)

**중요**: billyjo.co.kr 상세페이지에 등록된 정보는 **본사(제조사 또는 정식 유통사)가 직접 빌리조에 제공한 공식 자료**이다. 따라서:

- 빌리조 페이지에 명시된 스펙·기능·렌탈 조건은 confidence ≥ 0.85 부여
- 공식 사이트에 없고 빌리조에만 있는 정보도 신뢰 (단일 출처 검수 큐 트리거 면제)
- 빌리조 정보와 공식 사이트가 다르면 **빌리조 우선** (본사가 빌리조에 최신 정보를 제공했을 가능성)

크롤 순서: 1) 빌리조 상세페이지 → 2) 제조사 공식 → 3) 가격비교 (다나와).

**제조사 공식 페이지 추출 방법**: 상세페이지 전체가 아니라 **페이지 하단의 "제품 스펙 표"를 우선 참조**한다. 보통 `<table>` 또는 `<dl>` 형태로 모델명·브랜드·방식·타입·규격·관리주기·렌탈정보 등이 정리되어 있음. 본문 이미지·마케팅 텍스트는 보조 참조만 (표현 모호·과장 가능성 + 추출 정확도 낮음).

## 9. 검토 큐 운영 가이드

### 9.1 검토 필요로 분류되는 케이스
- AI 추출 신뢰도 < 0.7
- 핵심 항목(살균 기능, 필터 종류 등)이 단일 출처에서만 확인됨
- 제조사 공식 페이지에서 명시되지 않은 값을 다나와 등 2차 출처에 의존
- 카테고리 라우팅이 모호한 경우 (예: "에어 워셔" — 공기청정기 vs 가습기)

### 9.2 직원 처리 절차
1. 검토 큐에 들어온 제품 카드 미리보기
2. 빨간색·노란색 플래그 항목 확인
3. 제조사 콜센터/공식 카탈로그 PDF로 1차 확인
4. 확인된 값 입력 → AI가 점수 재계산
5. 승인 후 노출

### 9.3 룰북 갱신 사이클
- **월 1회**: 출처 화이트리스트 유효성 점검 (사이트 구조 변경 대응)
- **분기 1회**: 사은품 패턴 재학습 (브랜드별 프로모션 데이터 누적)
- **반기 1회**: 페르소나 사전 갱신 (실제 구독자 DB 분포와 비교, 신규 페르소나 추가/통합)
- **연 1회**: 패밀리 분류 검토 (신규 제품 카테고리 등장 시 패밀리 추가 또는 분리)

## 부록 A. 점수 → 등급 변환 (v0.2)

각 패밀리의 점수 산정 규칙은 그대로 유지하되, **사용자 노출 시에는 letter 등급으로 변환**한다. 점수 원본은 DB에 저장만 하고 카드 표면에는 노출 금지.

### A.1 매핑표 (Phase 1 — 절대 매핑)

| 점수 | letter | 한글 라벨 (개별 지표) | 컬러 | 의미 |
|------|--------|---------------------|------|------|
| 90-100 | **S** | 최고 | 🔵 `#0838F8` 파랑 | 카테고리 최상위 |
| 85-89 | **A+** | 적극추천 | `#1a87ac` 파랑-초록 | 우수+ |
| 80-84 | **A** | 추천 | 🟢 `#16a34a` 초록 | 우수 |
| 70-79 | **B+** | 우수 | `#4ec727` | 양호 |
| 60-69 | **B** | 좋음 | 🟢 `#84cc16` 연두 | 보통 (잘 작동) |
| 50-59 | **C+** | 적합 | `#c2ce15` | 보통 하위 |
| 40-49 | **C** | 보통 | 🟡 `#facc15` 노랑 | 개선 필요 |
| <40 또는 confidence<0.7 | **평가 없음** | (별도 텍스트, letter 사용 금지) | ⚪ `#999` 회색 | 데이터 부족 |

### A.2 종합 평가 산정

```python
def composite_grade(metrics: dict) -> tuple[str, float | None]:
    """4지표 등급 → 종합 등급. '평가 없음' 지표는 제외."""
    valid_scores = [m['score'] for m in metrics.values()
                    if m['grade'] != '평가 없음']
    if len(valid_scores) < 2:
        return ('평가 없음', None)  # 너무 적으면 종합 불가
    avg = sum(valid_scores) / len(valid_scores)
    return (score_to_letter(avg), round(avg, 1))
```

- **소수점 표기**: 게이지 중앙에 letter (큰 글씨) + "≈ {평균소수점}" (보조)
- **부분 평균 표기**: 일부 지표 평가 없음이면 "3지표 기준 · 2.0" 형식
- **종합 letter 범위**: S/A+/A/B+/B/C+/C (개별 지표와 동일). D 또는 평가 없음으로 산출되면 카드 자체 검수 큐.

### A.3 표시 형식

**종합 평가 (SVG 게이지)** — letter만:
```
종합 평가
   A          ← 큰 글씨 (font-size 26, Pretendard Bold)
≈ 2.0        ← 작은 글씨 (font-size 8.5, 회색)
```

**개별 지표 (4지표 배지)** — 한글 라벨 + letter 보조:
```html
<span class="grade-badge g1">최고<small>S</small></span>
<!-- 렌더링: [최고 | S] (세로 구분선으로 분리) -->
```

**평가 없음** — letter 미사용:
```html
<span class="grade-badge g-d">평가 없음</span>
<!-- 렌더링: [평가 없음] (회색 배경, ? 기호 없음) -->
```

### A.4 운영 Phase별 매핑 진화

| Phase | 카테고리당 누적 제품 | 매핑 방식 |
|-------|---------------------|---------|
| **Phase 1** (현재) | < 30 | 위 A.1 절대 매핑 (고정 곡선) |
| Phase 2 | 30-100 | 절대 매핑 + 카테고리 median 보정 (`±10점`) |
| Phase 3 | 100+ | percentile 기반 상대 매핑 (상위 5%=S, 상위 5-15%=A+ …) |

각 패밀리(F01~F14)가 독립적으로 Phase 전환. F01 정수가 Phase 3 진입해도 F09 건강운동은 여전히 Phase 1일 수 있음. Phase 전환 시 기존 등급 일괄 재산정 (검수 큐 경유).

### A.5 검증 사례 (F01)

| 모델 | 4지표 점수 | 등급 변환 | 종합 |
|------|-----------|----------|------|
| 코웨이 CHP-7211N | 88·92·96·95 | A+·S·S·S | **S** (≈92.8) |
| 세스코 EWBD351 | 90·80·평가없음·65 | S·A·평가없음·B | **A** (3지표 기준 · 78.3) |
