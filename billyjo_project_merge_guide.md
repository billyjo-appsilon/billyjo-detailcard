# 프로젝트 통합 가이드 — 충돌 점검 & 디자인 통일

> **상황**: 본 프로젝트(빌리조 AI 상세페이지 룰북 작업)를 "BillyJo 빌리조 오토메이션" 프로젝트로 통합.
>
> **목적**: (1) 통합 시 발생 가능한 충돌 점검 (2) 이 프로젝트의 디자인·구조를 단일 기준(single source of truth)으로 통일.
>
> **사용법**: 통합 작업자가 양쪽 프로젝트 산출물을 비교하면서 본 문서 체크리스트로 결정. 충돌 항목은 본 문서의 "이쪽 기준" 규칙을 따른다.

## 1. 카드의 페이지 내 배치 사양 (확정)

### 1.1 배치 원칙

**제품 상세페이지의 "상품정보(상세설명 이미지)" 섹션 바로 위 = 본 카드 위치**

기존 상단 영역(제품 이미지 갤러리·렌탈료/카드할인 박스·기본정보·렌탈사 비교 등)은 그대로 유지하고, 사용자가 페이지를 스크롤하면서 구매 결정에 진입하기 직전 시점에 본 카드가 노출된다. 카드 아래에는 기존 상품정보(상세설명 이미지)·고객센터·렌탈 주문혜택이 이어진다.

> 카드를 페이지 최상단으로 올리지 말 것 — 사용자가 가격·옵션을 보고 관심을 결정한 직후 단계에서 노출되어야 4지표·페르소나·사은품 정보의 임팩트가 크다.

**CSS 정리** (v0.2 — 제거 대상 박스와 동일 가로 + 마진 제로):
```css
#ai-card-root {
  display: block;
  width: 50%;
  margin: 0 0 0 auto;              /* 마진 모두 0, 우측 정렬은 auto */
  padding: 0 15px 20px;            /* .prod_table_wrap의 좌우 15px와 동일 */
  box-sizing: border-box;
}
@media (max-width: 1279px) and (min-width: 1024px) {
  #ai-card-root { padding-left: 25px }   /* .right padding-left:10px 보정 (15+10) */
}
@media (max-width: 1023px) {
  #ai-card-root { width: 100%; margin: 0; padding: 0 15px 20px }
}
@media (max-width: 600px) {
  #ai-card-root { padding: 12px 0 0 0; margin: 0 }
}
```

**왜 마진 0인가**:
- 카드 폭과 위치를 정확히 `.prod_table_wrap`(제거 대상)과 매칭하려면 외부 margin도 동일해야 한다.
- 카드와 빌리조 콘텐츠 사이의 수직 여백은 카드 자체 그림자·테두리로 시각 분리. `margin-top/bottom`으로 띄우면 정렬이 어긋남.

- **데스크톱**: 콘텐츠 영역 최대 폭 (헤더·메뉴 폭과 동일 정렬)
- **모바일**: 화면 전폭, 좌우 16-20px 패딩
- **상하 마진**: 카드 위쪽은 페이지 상단 헤더와의 간격 20-32px, 아래쪽은 다음 섹션과 24-40px

### 1.3 GTM 삽입 위치 사양

```html
<!-- 빌리조 상세페이지 템플릿 (실제: prod_view/<id>) -->
<div id="product-detail-page">
  <div class="prod_view_top">
    <!-- 기존: 이미지 갤러리, 렌탈료/카드할인 박스, 기본정보, 렌탈사 비교 등 -->
  </div>

  <!-- ↓↓↓ GTM이 여기에 카드 JSON을 바인딩해 삽입 (상품정보 섹션 직전) ↓↓↓ -->
  <div id="ai-product-card" data-product-id="{{product_id}}"></div>
  <!-- ↑↑↑ 상품정보 위, 구매 결정 직전 단계 ↑↑↑ -->

  <!-- 상품정보 -->
  <div class="prod_view_bot mt10">
    <div class="v_tit">상품정보</div>
    <div class="prod_view_detail">
      <!-- 기존 상세설명 이미지 -->
    </div>
  </div>
  <!-- 이하 고객센터·렌탈 주문혜택 등 -->
</div>
```

GTM 태그는 `#ai-product-card` 컨테이너를 찾아 카드 JSON으로 채운다. 이 컨테이너가 `<!-- 상품정보 -->` 섹션 바로 위에 위치하도록 빌리조 페이지 템플릿(`/html/dh_prod/prod_view/*`)을 수정해야 함.

### 2.1 즉시 결정이 필요한 항목 (High Risk)

다음 항목들은 양쪽이 다르면 **시스템 통합 자체가 안 되므로** 가장 먼저 확인:

- [ ] **2번 (점수 표시 방식)** — 시각적 통일성의 핵심
- [ ] **3번 (종합점수 계산)** — 점수가 달라지면 결과 자체가 달라짐
- [ ] **6번 (컬러 시스템)** — 토큰 변수가 다르면 카드가 깨짐
- [ ] **12번 (GTM 역할)** — 실시간 AI 호출이 설계되어 있다면 비용·속도 문제

### 2.2 점진적으로 통일 가능한 항목 (Low Risk)

다음은 일단 본 프로젝트 기준으로 노출하고, 운영하면서 조정:

- 페르소나 풀의 구체적 페르소나 정의 (페르소나는 데이터 누적되면 자동 조정)
- 사은품 예측 패턴 (브랜드별 패턴 학습 데이터)
- 페르소나 구성비 % (실제 구독자 DB 누적되면 정확해짐)

## 3. 디자인 통일 규칙 (본 프로젝트 기준)

### 3.1 컬러 토큰 — 통일 기준

본 프로젝트에서 사용된 빌리조 디자인 토큰을 그대로 단일 기준으로 채택:

| 토큰 | 용도 |
|------|------|
| `var(--color-text-info)` | 강조 색상 (점수 숫자, pill 활성 상태 텍스트, 게이지 점수 호) |
| `var(--color-background-info)` | 강조 배경 (pill 활성 상태 배경, DB 연동 placeholder 배지) |
| `var(--color-text-primary)` | 주 텍스트 (제품명, 페르소나 이름) |
| `var(--color-text-secondary)` | 라벨·보조 텍스트 (지표 라벨, 페르소나 설명) |
| `var(--color-text-tertiary)` | 메타 정보 (모델번호 라인, pill 비활성 텍스트) |
| `var(--color-background-primary)` | 카드 배경 (가장 바깥) |
| `var(--color-background-secondary)` | 내부 박스 배경 (스펙요약, 페르소나 카드, 사은품 박스) |
| `var(--color-border-tertiary)` | 카드 테두리, 섹션 구분선, pill 비활성 테두리 |
| `var(--border-radius-md)` | 내부 박스 모서리 |
| `var(--border-radius-lg)` | 카드 자체 모서리 |

**충돌 시 처리**: 다른 프로젝트의 카드가 hex 색상(#xxxxxx)이나 다른 변수명을 사용했다면, 위 토큰으로 일괄 치환.

| 종합점수 숫자 | 20px / weight 500 |

**충돌 시 처리**: 다른 프로젝트가 다른 크기·굵기를 썼다면, 위 규격으로 일괄 변경. 본문 폰트 패밀리는 **Pretendard 단일 사용**으로 통일 (3.2.1 참조). 빌리조 사이트 글로벌 폰트(Open Sans·NanumSquare)와 별개로 카드 영역만 Pretendard.

#### 3.2.1 폰트 패밀리 & Bold 처리 (절대 규칙 #11)

**본문 폰트**: 카드 영역 전체를 **Pretendard**로 통일. 빌리조 사이트 글로벌 폰트(Open Sans·NanumSquare)와 별개. 이유: 한국어·영문·숫자 혼용 시 정합성·자간·정렬이 가장 안정적이고, Bold weight 톤이 본문과 자연스럽게 연결됨.

**CDN 로드**:
```html
<link rel="stylesheet" href="https://cdn.jsdelivr.net/gh/orioncactus/pretendard@v1.3.9/dist/web/static/pretendard.min.css">
```

**본문 CSS**:
```css
body, .card { font-family: 'Pretendard', 'Apple SD Gothic Neo', '맑은 고딕', sans-serif; }
```

**Bold 처리**: 볼드 강조가 필요한 텍스트는 반드시 `font-weight:700`로 명시. `font-weight:bold` 또는 `<b>` 만으로 끝내지 말 것 — 일부 브라우저·OS에서 시스템 폰트로 폴백되어 톤이 깨질 수 있음.

```css
/* 강조 어휘 + 구조 앵커 라벨 모두 Pretendard Bold */
strong, .badge-best, .m.top-metric .ms,
.rec strong, .gift-v strong,
.ml, .sl, .sv,
.sec-t, .step-h, .p-t {
  font-family: 'Pretendard', sans-serif;
  font-weight: 700;
}
```

**Bold 적용 대상 매트릭스**:

| 종류 | 셀렉터 | 예시 |
|------|--------|------|
| 강조 어휘 | `<strong>` | 제품 주요 강점, 가격 경쟁력, 핵심 사은품 |
| 최고점 메트릭 | `.m.top-metric .ms` | 4지표 중 가장 높은 점수 |
| 배지 | `.badge-best` | "최고점", "BEST" 라벨 |
| **4지표 라벨** | `.ml` | 가격대, 보관성능, 에너지·위생, AI·편의기능 |
| **6스펙 라벨/값** | `.sl`, `.sv` | 가격대 / 200만원대, 용량 / 875L 등 |
| **섹션 제목** | `.sec-t` | 기본정보, 구독자 Top3, 예상 최대 지원금 |
| **Step 제목** | `.step-h` | 정수성능, 위생관리, AI·편의기능 |
| **페르소나 이름** | `.p-t` | 30평대 4인 가족, 알러지·호흡기 민감 가족 |

**금지**: `<b>`, `font-weight: bold` (선언만으로 끝낸 경우), `<strong>` 태그를 그냥 두고 폰트 미지정.
- **적용 대상** (어떤 정보를 Bold로 강조할지):
  | 영역 | 강조 대상 예시 |
  |------|---------------|
  | 추천 가이드 (.rec) | 제품 주요 강점, 차별화 기술 (예: "875L 매직스페이스 4도어", "메탈쿨링·도어쿨링+") |
  | 가격 경쟁력 | 렌탈료, 약정 기간, 할인 금액 (예: "월 17,900원", "월 최대 25,000원 청구할인") |
  | 사은품·보증 (.gift-v) | 핵심 혜택 (예: "초기비용 0원", "컴프레서 10년 평생보증", "LG 케어솔루션 방문관리") |
  | 메트릭 최고점 (.m.top-metric .ms) | 가장 높은 4지표 점수 |
  | 배지 (.badge-best) | "최고점", "BEST" 라벨 |
- **금지**: `<b>`, `font-weight: bold` (선언만으로 끝낸 경우), `<strong>` 태그를 그냥 두고 폰트 미지정.

## 5. 통합 후 작업하지 말 것 (반드시 폐기)

다음은 본 프로젝트 기준과 충돌하는 패턴이며, 통합 시 폐기:

- ❌ 카드를 페이지 최상단·사이드바·고객센터 아래 등 임의 위치에 배치 (반드시 "상품정보" 섹션 바로 위 + 우측 정렬)
- ❌ 카드에 외부 `margin-top/bottom/left` 적용 (margin 모두 0, 우측 정렬은 margin-right가 아닌 `margin-left: auto`로)
- ❌ 카드 padding을 `0 20px 20px` 같은 임의값으로 설정 (반드시 `.prod_table_wrap` 좌우 15px와 동일하게)
- ❌ 카드 라벨에 "AI", "자동생성", "AI Generated" 등 내부 용어 노출 (반드시 "📊 빌리조 제품분석")
- ❌ 게이지 중앙에 "≈ 2.0", "3지표 기준" 같은 소수점·산출 디테일 표시 (letter만 노출)
- ❌ 카드 식별 배지(`.ai-card-tag`)를 `top: -10px` 같은 큰 음수로 띄움 (반드시 `top: -1px`로 카드에 거의 붙임)
- ❌ 모바일에서 4지표 카드(`.m`)를 column으로 두 줄로 표시 (반드시 row 1줄)
- ❌ 카테고리 메뉴(`.category__wrap`)에 반투명 fade 그라데이션(`linear-gradient(transparent → #fff)`) 사용 (위 버튼을 가림. ▼ 토글만 사용)
- ❌ 카테고리 메뉴를 3줄 이상으로 표시 (max-height로 2줄 제한)
- ❌ 빌리조 원본 `/css/common.css` 직접 수정 (반드시 오버라이드 `<style>` 블록 또는 별도 파일로 cascade)
- ❌ 점수를 별점(★) 또는 막대 또는 숫자로 직접 표시 (반드시 letter 등급)
- ❌ 등급에 letter "D" 사용 (반드시 "평가 없음" 텍스트 + 회색 배지)
- ❌ 종합 평가에 한글 라벨 표시 (종합은 letter만, 개별 평가는 한글+letter)
- ❌ 종합점수에 가중평균 적용 (단순평균 + `평가 없음` 제외)
- ❌ 페르소나 Top 1 또는 Top 5+ (정확히 Top 3)

## 6. 어시스턴트가 직접 비교할 수 없는 부분 (한계 명시)

작업자가 직접 확인해야 할 부분:

1. **BillyJo 오토메이션 프로젝트의 실제 지식 베이스 내용** — 어시스턴트 검색 범위는 현재 작업 프로젝트로 한정.
2. **이미 운영 중인 상세페이지 디자인의 시각적 모양** — 스크린샷·실제 페이지 비교 필요.
3. **빌리조 디자인 시스템 변수의 실제 색상값** — `var(--color-text-info)` 등이 실제로 어떤 색으로 렌더링되는지는 빌리조 CSS에 정의됨.
4. **개발팀이 이미 만든 컴포넌트·코드 자산** — 코드 충돌 분석은 별도 작업.

이 4가지를 채팅에 공유해주시면, 그 시점부터는 어시스턴트가 항목별로 비교·판정 가능.
